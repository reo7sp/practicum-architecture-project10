# 5. Разработка cutover-плана

## Выбор стратегии

Выбранный вариант: Parallel run.

Плюсы:

1. Позволяет запустить новый контур рядом со старым и сравнивать результаты.
2. Снижает риск потери данных при переходе с Excel, файлов и 1С.
3. Даёт время на проверку ролей доступа, интеграций и качества миграции.
4. Позволяет откатиться на старый контур без полной остановки бизнеса.

Минусы:

1. Требует временного двойного учёта.
2. Повышает нагрузку на команду и инфраструктуру.
3. Требует отдельного контроля синхронизации данных.


## Cutover-план

| Этап | Ключевые задачи | Ответственные | Сроки | Риски |
|---|---|---|---|---|
| 1. Расстановка приоритетов миграции | Определение критических процессов и доменов миграции | Техдиректор, владельцы процессов | За 4-6 недель до cutover | Распыление ресурсов |
| 2. Подготовка данных, среды и команды | Очистка и дедупликация данных, подготовка среды, обучение пользователей | Техдиректор, администратор сети, бизнес-аналитик | За 3-4 недели до cutover | Низкое качество исходных данных |
| 3. Тестирование миграции и пилотный прогон | Тестовая миграция, проверка интеграций, пилотный прогон | Техдиректор, администратор сети | За 2-3 недели до cutover | Нестабильность интеграций |
| 4. Подготовка переключения между контурами | Подготовка freeze-окна, правил маршрутизации и go / no-go чек-листа | Техдиректор, администратор сети | За 1-2 недели до cutover | Ошибки в доступах и ролях |
| 5. Реализация parallel run | Пилотная группа, parallel run, сверка результатов двух контуров | Ресепшен, врачи, кассир, бухгалтерия, склад | 1-2 недели | Расхождение данных между старым и новым контуром |
| 6. Мониторинг после переключения | Контроль метрик, алертов, критических сценариев и инцидентов | Техдиректор, администратор сети, ключевые пользователи | Первые 5-10 рабочих дней | Деградация производительности нового контура |
| 7. План отката на старый контур | Возврат трафика и операций в старый контур при критическом сбое | Техдиректор, администратор сети | В cutover-окно | Потеря части операций |



## Риски, меры и распределение ролей

| Риск | Мера управления | Ответственные |
|---|---|---|
| Распыление ресурсов | Фиксация приоритетных доменов и ограничение объёма первой волны миграции | Техдиректор, владельцы процессов |
| Низкое качество исходных данных | Очистка, дедупликация, нормализация и предварительная валидация данных | Техдиректор, бизнес-аналитик |
| Нестабильность интеграций | Выполнение интеграционного прогона, буферизация обмена, внедрение идемпотентности | Техдиректор, администратор сети, бухгалтерия |
| Ошибки в доступах и ролях | Подготовка матрицы ролей, выполнение сценариев проверки доступа, пилотный запуск | Техдиректор, администратор сети |
| Расхождение данных между старым и новым контуром | Определение единого источника правды, ведение журнала расхождений, ежедневная сверка | Техдиректор, бизнес-аналитик |
| Ошибки пользователей при переходе | Подготовка инструкций, обучение пользователей, запуск пилотной группы | Бизнес-аналитик, ресепшен, врачи, кассир |
| Деградация производительности нового контура | Проведение нагрузочного тестирования, резервирование мощности, настройка метрик и алертов | Техдиректор, администратор сети |
| Потеря части операций | Журнал изменений в cutover-окно и заранее определённые критерии rollback | Техдиректор, администратор сети |
